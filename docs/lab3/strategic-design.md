## 1. Виявлення подій та команд (Event Storming Lite)

Аналіз бізнес-процесів платформи GamingStore дозволив виділити наступні ключові події (Domain Events), команди (Commands) та згрупувати їх навколо агрегатів (Aggregates), які відповідають за консистентність даних.

| Агрегат                          | Відповідальність                                     | Команди (Commands)                                           | Події (Domain Events)                                                 |
| :------------------------------- | :--------------------------------------------------- | :----------------------------------------------------------- | :-------------------------------------------------------------------- |
| **ShoppingCart** (Кошик покупок) | Формування замовлення до моменту фіксації ціни.      | `AddProductToCart`<br>`RemoveProductFromCart`<br>`ClearCart` | `ProductAddedToCart`<br>`ProductRemovedFromCart`<br>`CartCleared`     |
| **Order** (Замовлення)           | Фіксація наміру придбати товари за узгодженою ціною. | `PlaceOrder`<br>`CancelOrder`                                | `OrderPlaced`<br>`OrderCancelled`                                     |
| **Payment** (Оплата)             | Фінансові транзакції та їхні статуси.                | `RequestPayment`<br>`ProcessPayment`<br>`FailPayment`        | `PaymentRequested`<br>`PaymentSucceeded`<br>`PaymentFailed`           |
| **Inventory** (Складські запаси) | Контроль наявності фізичних копій ігор чи консолей.  | `ReserveStock`<br>`DeductStock`<br>`ReleaseStock`            | `StockReserved`<br>`StockDeducted`<br>`StockReleased`<br>`OutOfStock` |
| **Shipment** (Доставка)          | Логістичний процес після успішної оплати.            | `PrepareShipment`<br>`DispatchShipment`<br>`DeliverShipment` | `ShipmentPrepared`<br>`ShipmentDispatched`<br>`ShipmentDelivered`     |

## 2. Обмежені контексти (Bounded Contexts)

Для забезпечення балансу між чистотою архітектури та складністю технічної реалізації, систему декомпоновано на 3 обмежених контексти. Розділення проведено на основі лінгвістичних та процесних розривів.

| Контекст                                          | Зона відповідальності                                                                              | Евристики розділення меж                                                                                                                                                                     |
| :------------------------------------------------ | :------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Catalog Context**<br>(Каталог та Вітрина)       | Управління асортиментом товарів, їхніми описами, цінами та категоріями. Забезпечує роботу вітрини. | **Відповідальні особи:** Контент-менеджери та маркетологи.<br>**Термінологія:** `Product` — це сутність із картинками, текстом та базовою ціною.                                             |
| **Ordering Context**<br>(Управління замовленнями) | Робота з кошиком, оформлення покупки (Checkout), фіксація фінальної ціни та обробка оплати.        | **Розрив у часі:** Оформлення та оплата відбуваються синхронно ("тут і зараз").<br>**Термінологія:** `Product` перетворюється на `OrderLine` із жорстко зафіксованою ціною.                  |
| **Fulfillment Context**<br>(Виконання та Склад)   | Фізичне управління залишками товарів, пакування та відправка замовлення клієнту.                   | **Розрив у часі:** Збирання та доставка відбуваються пізніше.<br>**Відповідальні особи:** Комірники та логісти.<br>**Термінологія:** `Product` — це `StockItem` (лише артикул та кількість). |

### Діаграма взаємодії (Context Map)

```mermaid
graph TD
    Catalog[Catalog Context]
    Ordering[Ordering Context]
    Fulfillment[Fulfillment Context]

    Ordering -->|Читає ціни та описи| Catalog
    Ordering -->|Ініціює збирання та відправку| Fulfillment
```

## 3. Єдина мова (Ubiquitous Language)

Нижче наведено словники термінів для кожного обмеженого контексту. Головна особливість цієї моделі — концепція «Товару» (Product), яка кардинально змінює своє значення та набір атрибутів залежно від того, в якому бізнес-процесі вона бере участь.

### 3.1 Catalog Context (Каталог та Вітрина)

> Бізнес мислить категоріями маркетингу та привабливості для клієнта.

| Термін                   | Бізнес-значення                                                                                               |
| :----------------------- | :------------------------------------------------------------------------------------------------------------ |
| **Product** (Товар)      | Одиниця асортименту на вітрині. Містить маркетинговий опис, скріншоти, відео, технічні вимоги та базову ціну. |
| **Category** (Категорія) | Логічне групування товарів (наприклад, "Шутери", "Консолі") для зручної навігації.                            |
| **Publisher / Brand**    | Компанія, яка створила або випустила товар.                                                                   |
| **Display Price**        | Вітринна ціна до застосування будь-яких знижок під час оформлення.                                            |
| **Tag** (Характеристика) | Додаткові мітки для фільтрації (наприклад, "Мультиплеєр").                                                    |

### 3.2 Ordering Context (Управління замовленнями та Продажі)

> Бізнес мислить категоріями грошей, домовленостей та фіксації умов.

| Термін                    | Бізнес-значення                                                                                             |
| :------------------------ | :---------------------------------------------------------------------------------------------------------- |
| **Order Line** (Рядок)    | Трансформація "Товару" в контексті продажів. Має зафіксовану ціну та кількість. Не має опису чи скріншотів. |
| **Order** (Замовлення)    | Зафіксований намір клієнта придбати обрані рядки за узгодженою фінальною ціною.                             |
| **Checkout** (Оформлення) | Процес перетворення попереднього кошика на підтверджене замовлення.                                         |
| **Final Price**           | Точна сума до сплати після застосування всіх знижок та вартості доставки.                                   |
| **Payment Status**        | Індикатор виконання фінансових зобов'язань (Очікує оплати, Сплачено, Відхилено).                            |

### 3.3 Fulfillment Context (Виконання та Склад)

> Бізнес мислить фізичними об'єктами, коробками, полицями та логістикою.

| Термін                 | Бізнес-значення                                                                                                |
| :--------------------- | :------------------------------------------------------------------------------------------------------------- |
| **Stock Item / SKU**   | Трансформація "Товару" в контексті складу. Фізична коробка з артикулом, вагою, габаритами та місцем на полиці. |
| **Available Quantity** | Кількість одиниць певного артикулу, готових до продажу новим клієнтам.                                         |
| **Reserved Quantity**  | Одиниці товару, які фізично є на складі, але вже обіцяні іншому клієнту.                                       |
| **Shipment / Package** | Фізична коробка, у яку спаковані складські одиниці, готова до передачі кур'єрській службі.                     |
| **Delivery Address**   | Фізичні координати або відділення пошти для доставки посилки.                                                  |
